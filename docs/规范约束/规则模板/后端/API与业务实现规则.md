# API 与业务实现规则

## 触发条件

修改接口入口、请求响应、业务服务、鉴权、校验、权限、异常、日志、状态流转、金额、时间、并发或幂等逻辑时必须阅读。

## 先读与复用优先

1. 先阅读同类接口入口、业务服务、请求响应类型、错误表达、日志、权限和测试，并确认完整调用链。
2. 优先复用当前项目已有的鉴权、校验、转换、异常、错误码、上下文、状态和幂等能力；至少阅读一个真实成功或失败示例。
3. 新增业务边界、契约类型或公共基础能力前，先说明现有能力无法扩展的真实原因。

## 强制规则

1. API 入口只承担项目现有框架允许的鉴权、校验、转换和调用，不把业务逻辑塞进入口层。
2. 业务实现复用现有错误码、异常基类、日志格式、转换器和上下文；禁止裸抛不受控异常或返回伪成功。
3. 修改契约时同步所有调用方、类型、测试与 API 文档；兼容策略必须明确到字段和版本。
4. 状态、金额、时间、精度、并发和幂等按项目已有模型处理，不发明新的表达方式。
5. 日志记录定位所需关键参数并按项目规则脱敏，禁止输出密钥、令牌和完整敏感字段。

## 禁止事项

- 禁止将业务编排、持久化细节或外部调用塞进项目既有边界之外的入口层。
- 禁止裸抛不受控异常、捕获后伪成功、以默认值掩盖校验失败，或根据名称猜测错误码、权限和状态语义。
- 禁止只改接口一侧；请求响应、调用方、类型、权限、测试和接口资料必须按真实影响面同步检查。
- 禁止凭常见做法定义金额单位、时间格式、精度、锁、事务、状态机或兼容策略。

## 验证要求

- 验证正常、校验失败、权限失败、异常、状态边界和直接相关历史行为。
- 涉及契约时核对调用链各端的请求、响应、错误和兼容行为；涉及敏感信息时检查日志输出。
