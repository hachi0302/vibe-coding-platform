# 前端验证规则

## 触发条件

修改前端逻辑、组件、页面、路由、请求、类型、样式、构建配置或修复前端缺陷后必须阅读。

## 先读与复用优先

1. 先查看当前项目已有的单元测试、组件测试、端到端测试、构建命令和同类页面验证方式。
2. 优先在已有测试文件、测试工具和测试数据组织中补充用例；没有真实测试入口时记录待补信息，不伪造测试命令。

## 最低验证

| 变化 | 必须验证 |
|---|---|
| 纯逻辑/hook | 单元测试：正常、边界、异常 |
| 组件 | 组件测试：渲染、交互、禁用/加载/错误状态 |
| 页面/路由 | 关键用户路径、权限、刷新/返回、空态与失败态 |
| API/类型 | 类型检查、请求参数、响应映射、错误处理 |
| 样式/布局 | 目标视口、长文本、空数据、主题/响应式（项目支持时） |

必须运行项目真实命令：`{{类型检查命令}}`、`{{lint命令}}`、`{{测试命令}}`、`{{构建命令}}`。不存在的命令从表中删除，不伪造。

## 禁止事项

- 禁止只确认页面能渲染，而遗漏加载、空态、失败、权限、禁用、返回或刷新等已存在行为。
- 禁止把快照、模拟调用次数或手工描述当作真实交互和请求契约已经验证的证据。
- 禁止编写或执行当前项目未配置的命令，并把结果表述为项目验证。

## 验证要求

- 按改动类型执行表中适用的真实验证；修复缺陷时必须覆盖原问题和至少一个直接相关回归场景。
- 记录执行命令、退出码、关键结果和无法执行的原因；样式或布局变化要使用项目实际支持的视口、主题或设备条件验证。
