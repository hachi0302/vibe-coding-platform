# 事实与兜底边界

## 触发条件

定位缺陷、设计兼容方案、处理异常、读取配置、解析外部数据、映射枚举或处理信息缺失时必须阅读。

## 先读与复用优先

- 先读取真实代码、日志、请求响应、数据结构、配置说明和已有成功链路，确认项目已经采用的失败表达与异常处理方式。
- 需要补充字段、映射或错误处理时，先查现有契约、枚举、校验器和调用方；无法确认的内容进入待补信息。

## 结论标准

- “根因”必须由代码、日志、数据库、请求响应或稳定复现中的充分证据闭环；证据不足只能写“推测”。
- 访问不到的日志、数据库或外部系统必须明确写“未配置/不可访问”，不得假装查询过。
- 配置文件、配置键、依赖或部署清单只证明存在接入线索，不证明能力可用；只有安全最小探测成功并留存脱敏证据后才可标记“可用”。
- 可以仅靠代码确认的问题直接给结论，不强制无关的日志或数据库步骤。

## 禁止事项

禁止在需求和项目证据未定义时自行加入：

- 默认值覆盖缺失必填字段；
- 吞异常、记录日志后返回成功；
- 未知枚举自动映射到正常值；
- 失败自动降级为旧逻辑或模拟成功；
- 多来源字段“谁有值用谁”的优先级猜测；
- 为通过测试而放宽权限、校验、状态或一致性约束。

禁止擅自兜底；正确做法是暴露真实错误、说明缺失配置或请求用户确认业务选择。

## 验证要求

- 验证异常、未知值、缺失配置和失败链路按真实约束可见，且不会被误报为成功。
- 修复问题时保留可复现的失败场景，并确认正常路径没有被兜底逻辑改变。
