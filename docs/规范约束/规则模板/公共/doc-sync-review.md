# 提交前文档一致性审核

## 触发条件

准备执行 `git commit`，或用户明确要求提交代码时，必须先执行本审核。

## 审核流程

1. 读取 `git diff --cached --name-status` 与完整暂存区 diff，确认本次真实改动。
2. 逐项判断是否影响业务功能、API/回调、数据表与字段、枚举、系统架构或公共能力。
3. 只更新确实受影响的长期文档；不得为了通过审核编造内容，也不得复制空模板充数。
4. 将本次需要同步的文档加入暂存区；无需更新时，保留“已审核且无文档影响”的证据结论。
5. 执行 `.claude/skills/doc-sync-review/scripts/doc-sync-gate.sh --record` 记录当前暂存区指纹，然后才可提交。
6. 指纹只对应当前暂存区；任何代码或文档重新暂存后都必须重新审核并记录。

## 项目长期文档

初始化时由平台根据前端、后端或全栈类型补充当前项目真实路径。不存在的文档类型不要求创建。
