# 开发流程与文档同步

## 新需求

1. 读业务功能总览，确认落点和可复用能力。
2. 先从项目发布/迭代事实确定 `v{版本}`，再扫描该版本目录，按项目现有位数取未占用的下一序号；详设、进度、前端接入和自测归档使用同一任务序号。禁止写死模板编号、复用已有编号或凭空创建版本。
3. 用项目详设模板创建本迭代详设；用户确认设计后才改业务代码。
4. 创建进度文档并实时记录用户反馈、测试和阻断。
5. TDD：失败用例 → 最小实现 → 正常/边界/异常/历史回归。
6. 同步所有受影响长期文档，业务功能总览每次功能变化必审视。

## Bug

1. 从症状与错误数据流入手复现，检索全部相同模式。
2. 区分根因和推测，列出代码/日志/数据证据。
3. 用户确认方案后做最小修复，先保留能失败的原 Bug case。
4. 运行原 Bug、边界、异常和相关历史回归；确认无同类遗漏。

## 代码优化

先证明当前问题和收益，保持外部行为不变；无法用测试锁定行为时不进行大范围重构。
