# 开发流程与文档同步

## 触发条件

开始新需求、缺陷修复、兼容性调整、性能优化或需要更新长期资料时必须阅读。

## 先读与复用优先

1. 先阅读当前项目已存在的业务、架构、接口、物理模型、枚举、前端或测试资料，确认本次改动不是重复建设。
2. 从真实代码中找到同类已完成实现，复用其任务拆分、调用链和验证方式；资料缺失时只记录待补信息。
3. 只有项目存在版本化资料目录和相应模板时，才按现有归档规则创建版本化资料。

## 新需求

1. 读业务功能总览，确认落点和可复用能力。
2. 根据当前项目已有的发布、迭代和归档证据确定资料位置；没有该机制时不得凭空创建版本编号、序号或目录规范。
3. 项目已有详设、进度或前端接入模板时按其结构创建；没有模板时只记录需求边界、影响面与待确认事项。
4. 用户确认设计边界后再改业务代码，并实时记录用户反馈、测试和阻断。
5. 使用项目现有测试方式覆盖正常、边界、异常和直接相关历史行为。
6. 审视并同步受影响长期资料，不能用空模板或审计说明替代真实资料。

## Bug

1. 从症状与错误数据流入手复现，检索全部相同模式。
2. 区分根因和推测，列出代码/日志/数据证据。
3. 用户确认方案后做最小修复，先保留能失败的原 Bug case。
4. 运行原 Bug、边界、异常和相关历史回归；确认无同类遗漏。

## 代码优化

先证明当前问题和收益，保持外部行为不变；无法用测试锁定行为时不进行大范围重构。

## 禁止事项

- 禁止跳过现有资料和同类实现，直接按常见做法新建框架、目录、接口或模型。
- 禁止在没有项目证据时强制要求版本化详设、固定编号、固定测试流程或特定开发方法。
- 禁止为了文档完整而臆造接口、枚举、表、页面、流程、历史问题或验证结论。

## 验证要求

- 对照实际 diff 检查受影响的代码、契约、测试和长期资料是否同步；没有影响时明确记录“已审视，无需更新”。
- 对需求或缺陷的关键路径执行当前项目可用的真实验证。
