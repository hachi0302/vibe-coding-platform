## 开发流程

1. 读业务功能总览和命中模块的架构文档。
2. 读 `.claude/rules/README.md`，按任务与代码 pattern 加载规则。
3. 新需求先写详设与进度文档；Bug 先复现并给出证据支持的根因或推测。
4. 先写失败测试，再做最小实现；覆盖正常、边界、异常、原 Bug 和相关历史回归。
5. 同步受影响的长期文档，向用户汇报真实命令和结果。
6. 提交前必须使用 `doc-sync-review` 审核暂存区与长期文档一致性，并记录当前暂存区审核凭证。
7. 提交与推送由用户选择；未经授权不执行。

## 工作纪律

- 改文件前先读文件、相关规则和同类实现。
- 优先复用已有组件、工具、模型、请求封装、错误处理和测试夹具。
- 只改当前任务直接相关内容，不覆盖用户未提交改动，不顺手重构。
- 结论必须来自代码、配置、测试、数据或用户材料；证据不足时明确写“推测”。
- 不添加未经需求或项目证据确认的默认值、降级成功、吞错或兜底分支。
- 修改公共能力、契约、枚举、数据结构前，全局检索所有调用方并回归。
