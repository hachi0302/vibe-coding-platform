## 待开发者补充

{{数据库能力补充表}}

在获得授权后，先按数据库方言完成最小只读探测。至少执行并记录脱敏结果：

```sql
SELECT 1;
```

还需确认当前数据库、当前用户、数据库版本、只读状态和授权范围。MySQL 可使用 `SELECT DATABASE(), CURRENT_USER(), @@version, @@read_only;` 与 `SHOW GRANTS;`；其他数据库使用等价只读语句。探测失败或无法证明只读身份时，状态保持“有证据但需配置”。

## 执行流程

1. 明确环境、业务号/主键、时间范围、目标表和最少所需字段。
2. 读取物理模型、Repository/DAO 与迁移脚本，确认表字段和关联关系。
3. 验证连接是否为已授权只读身份；只允许 `SELECT`、`WITH ... SELECT`、`SHOW`、`DESCRIBE`、`EXPLAIN`。
4. 先查主记录，再按真实外键或业务键追关联记录；限制字段、条件和行数。
5. 将查询结果与代码分支、状态转换和日志时间线交叉验证。
6. 输出脱敏 SQL、结果摘要、代码证据和结论可信度。

禁止执行 INSERT、UPDATE、DELETE、MERGE、DDL、存储过程、锁表或会改变会话/数据状态的语句。生产环境即使只读也必须遵守项目额外授权规则。

## 完成 Gate

- [ ] 数据库类型/版本与“环境 → 数据库”映射已确认。
- [ ] 凭证来源只记录名称和获取方式，不在 skill、文档或日志中保存秘密值。
- [ ] CLI/MCP/只读脚本真实存在，当前 Agent 可执行。
- [ ] 已验证只读账号和权限边界，不允许写入、DDL、存储过程或锁表。
- [ ] `SELECT 1` 与身份/版本/只读状态/授权范围探测成功，真实命令退出码为 0。
- [ ] 已记录验证时间、环境和脱敏证据；未全部满足时状态仍为“有证据但需配置”。

诊断任务本身还必须满足：查询目标、环境、条件、字段和行数均为解决问题所必需；SQL 已确认只读；结果可关联到真实代码/日志证据；不得把缺失记录、空值或连接失败自行解释为业务成功/失败。

## 失败处理

- 未授权或连接失败：保留脱敏错误与接入条件，不尝试绕过权限。
- 物理模型与运行库不一致：记录差异并停止猜字段，不执行修复 SQL。
- 查询结果不足以闭环：标记为当前推测，列出下一步最小只读取证。
