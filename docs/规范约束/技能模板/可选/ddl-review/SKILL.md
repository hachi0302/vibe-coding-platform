## 执行流程

1. 对照 `{{物理模型文档}}`、实体、查询和历史迁移确认真实需求。
2. 检查类型、长度、精度、默认值、NULL、主键、唯一约束、索引及命名。
3. Grep 所有读写方，确认新增、废弃、重命名和兼容窗口。
4. 评估锁表、全表扫描、数据回填、重复执行、上线顺序与回滚可行性。
5. 使用项目既有迁移工具和版本规则，不直接修改已发布迁移。
6. 更新物理模型总览并给出验证 SQL/测试证据。

输出按阻断、重要、建议分级；数据量或数据库版本未知时明确风险，不编造结论。

## 完成 Gate

- DDL 与实体、读写方、接口契约和物理模型一致。
- 已检查兼容窗口、锁表/扫描风险、数据回填、上线顺序和回滚路径。
- 已使用项目真实数据库版本与迁移工具验证，不修改已发布迁移。
- 每个 finding 包含位置、影响、证据和最小修复方向。

## 失败处理

- 数据量、版本或执行计划不可访问：明确风险范围，不写“无影响”。
- 迁移无法在安全环境执行：保留静态审查结果和待验证清单。
- 发现历史迁移已发布：新增修正迁移，不回写历史文件。
