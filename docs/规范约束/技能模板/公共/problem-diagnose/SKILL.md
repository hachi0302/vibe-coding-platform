# 定位问题根因

## 触发条件

- 用户提供错误信息、异常数据、错误行为、日志、截图或可复现线索，并要求诊断。
- 未确认根因前只输出事实与当前推测；用户授权后才进入开发修复。

## 项目资源

| 证据源 | 当前项目入口 | 何时使用 |
|---|---|---|
| 项目规则/总览 | `{{Agent入口}}`、`{{项目总览路径}}` | 每次必读 |
| 源码与测试 | {{项目证据路径}} | 所有问题 |
| 日志 | {{日志来源说明}} | 运行时/时序问题 |
| 数据库 | {{数据库证据说明}} | 数据/状态问题 |

日志和数据库不是形式化必经步骤：代码证据能直接闭环的空指针、类型或分支错误可直接定位；需要运行时事实时若对应能力已配置则必须使用。

## 证据标准

根因必须形成“症状 → 入口 → 错误数据/状态流 → 代码位置 → 可复现结果”的证据链。证据不足时只能标注推测。

## 执行步骤（Pipeline）

1. 提取环境、时间、请求/业务号、预期、实际、错误信息和复现条件。
2. 从报告症状定位最窄入口，沿调用链追踪参数、状态和数据转换，不先做全库漫游。
3. 按需取证：
   - 明确代码异常（如空指针、类型错误）：先看堆栈与代码。
   - 数据/状态异常：结合代码与相关表记录。
   - 环境或时序异常：结合可访问日志；有 `backend-log-diagnose` 时必须使用。
4. Grep 相同模式，判断是单点问题还是同类缺陷。
5. 输出已确认事实、根因或推测、影响范围、修复方案和验证方案。
6. 用户确认后才改代码；改动进入 `developer` 流程。

### Gate 1 — 现象可核对

至少明确预期、实际、入口和可用标识；缺少环境/业务号不影响静态代码结论时可继续，确实影响定位时只询问必要信息。

### Gate 2 — 证据链闭环

只有当错误值/状态能沿真实调用链追到产生位置，并能解释当前现象，才写“已确认根因”。否则明确写“当前推测”、支持证据、反证和下一步取证。

### Gate 3 — 方案覆盖同类模式

Grep 所有相同实现，区分应一并修复与仅表面相似的调用点。修复方案必须包含原 Bug case、边界、异常和历史回归，不只改报错行。

## 输出格式

| 项目 | 内容 |
|---|---|
| 现象与复现 | 可核对的输入、步骤与结果 |
| 证据 | 日志/数据/代码位置及相互关系 |
| 结论 | `已确认根因` 或 `当前推测` |
| 影响面 | 同模式、数据、接口和历史行为 |
| 修复与验证 | 最小改动、原 case 和回归范围 |

## 禁止事项

禁止把“看起来可能”包装成根因；禁止为了让流程继续私自添加异常吞噬、默认成功或数据兜底。

## 可执行验证

- 用项目资源中真实存在的复现、测试、只读查询或日志检索方式验证；每项命令记录范围、退出码与脱敏结果。
- 根因结论必须能解释原始现象并由独立证据交叉支持；否则保持“当前推测”。

| 缺失信息 | 已确认事实 | 影响 | 下一步最小取证 |
|---|---|---|---|
| {{环境/时间/业务号/日志/数据/复现步骤}} | {{路径、堆栈或截图位置}} | {{无法闭环的假设}} | {{最小只读命令、文件或用户材料}} |

## 完成 Gate

- 结论标记为“已确认根因”或“当前推测”，没有混写。
- 所有证据可定位到日志片段、查询结果、代码或稳定复现。
- 影响面与同类模式已扫描。
- 修复前等待用户确认；用户确认后移交 developer 和对应 self-test。

## 失败处理

- 复现失败：记录已尝试输入、环境和差异，不把未复现写成不存在问题。
- 日志/数据库不可访问：说明缺失证据及影响，继续完成仍可由代码证明的部分。
- 多个假设均可能：给出各自支持证据、反证和下一步最小取证，不任意选一个当根因。
