# 编写详细设计

## 项目资源

| 资源 | 当前项目位置 | 使用时机 |
|---|---|---|
| Agent 入口 | `{{Agent入口}}` | 每次触发先读 |
| 项目总览 | `{{项目总览路径}}` | 确认业务落点与边界 |
| 详设模板 | `{{详设模板路径}}` | 生成正式详设 |
| 进度模板 | `{{进度模板路径}}` | 详设完成后生成进度 |
| 详设归档 | `{{详设目录}}` | 按项目版本规则归档 |

当前源码与测试证据：{{项目证据路径}}。不存在的代码层不写入详设，禁止为填满模板而编造接口、表或第三方能力。

## 核心原则

详设必须从需求材料与现有代码反推，结论可追溯、方案可开发、验收可执行。禁止只改名词套模板。

## 执行流程（Pipeline，不可跳步）

1. 读取 `{{项目总览路径}}`、业务总览、架构、接口/模型文档及相关规则。
2. 逐项打开用户提供的文字、图片、原型、日志、接口资料；列出已确认事实与待确认项。
3. 沿入口、调用链、状态、持久化、异步、前端交互检索现有实现，优先确认可复用能力。
4. 判断需求边界、兼容性、数据影响和历史回归面；缺陷类需求区分根因与推测。
5. 按 `{{详设模板路径}}` 写入 `{{详设目录}}`，同步创建 `{{进度模板路径}}` 对应进度文档。
6. 用代码路径、类型、接口、表、测试位置支撑关键设计；无法确认的写为待确认，不编造。
7. 完成详设 Gate 后再请求用户确认方案。

### Phase 1 — 素材与代码盘点

逐个打开用户材料；再从用户指出的业务入口追到调用链、状态、数据、异步和测试。输出材料证据表、已有能力与缺失信息。没有读完不得进入方案阶段。

### Phase 2 — 现状、根因与影响面

新需求说明现有能力为什么不满足；Bug 必须将“已确认根因”和“当前推测”分开。GATE：根因、兼容边界或不可逆数据含义无法确认时，停止定稿并只问最少必要问题。

### Phase 3 — 方案与文件级设计

优先复用项目已有框架、组件、工具、错误体系和测试基座。方案必须明确到现有模块、文件、入口、契约、状态/数据结果与不变行为，不能只写分层概念。

### Phase 4 — 测试与评审

先设计正常、边界、异常、原 Bug 和相关历史回归用例，再对照详设模板逐项自审。GATE：用例不能证明关键方案时，回到 Phase 3 修订。

### Phase 5 — 归档与交接

生成详设和进度文档，列出长期文档同步项。向用户呈现关键决策与待确认项；用户确认前不进入业务代码开发。

## 完成 Gate

- 需求、非目标和验收标准明确。
- 现状、影响面、复用点与兼容策略有代码证据。
- 前端、后端、数据、异步仅在真实涉及的层次出现。
- 正常、边界、异常、原 Bug case 与历史回归均可测试。
- 文档无空占位符、无伪造路径、无自创框架。

## 失败处理

- 材料不可读：列出具体文件与错误，不假装已读取。
- 代码与文档冲突：以可运行代码和正式契约为事实，记录冲突并等待业务含义确认。
- 缺外部系统信息：将字段/状态标为待确认，不自创默认值或降级路径。
- 评审不通过：保留意见、证据和修改记录，回到对应 Phase，不跳到开发。

## 停止条件

关键业务含义、资金/权限/状态语义或不可逆数据变更无法确认时，停止方案定稿并提出最少必要问题。
