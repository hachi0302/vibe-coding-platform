# 处理评审意见

## 触发条件

- 用户提供评审意见、评审记录或修改要求，且能定位被评材料。
- 评审意见不是事实来源；每条都必须与真实代码、契约、文档或测试交叉核对。

## 执行步骤（Reviewer + Pipeline）

1. 读取评审原文、被评材料、关联代码与项目规则。
2. 每条意见独立核对证据并分类：
   - `确认问题`：直接修正文档或代码，并记录修改位置。
   - `非问题`：给出代码、契约或事实依据，不迎合修改。
   - `信息不足`：列出缺失证据与最小确认问题。
3. 修改后复查上下文一致性、测试影响和长期文档同步。
4. 输出逐条处置表，禁止遗漏、合并或只回复结论。

### Gate 1 — 意见盘点完整

必须先给每条意见编号，建立“意见 → 原文位置 → 相关代码/契约”映射；无法定位原文的意见标为信息不足，不脑补。

### Gate 2 — 分类有证据

`确认问题` 至少有一项直接证据；`非问题` 必须说明现有契约为何正确；`信息不足` 必须列最少需要补充的事实。没有证据不能因评审方语气确定就判为问题。

### Gate 3 — 联动一致

改详设后同步检查进度、测试用例、前端接入、API/模型和业务总览；改代码后进入 developer/self-test 流程，禁止只改一处造成文档互相冲突。

| 序号 | 评审意见 | 分类 | 证据 | 处置 | 修改位置 |
|---|---|---|---|---|---|

评审意见不是事实来源；技术上不成立的意见必须说明原因。

## 禁止事项

- 禁止因评审方语气、数量或优先级标签而跳过事实核对。
- 禁止合并、遗漏或擅自扩大评审意见；禁止将信息不足项伪装成“已修复”。

## 可执行验证

- 每个确认问题应有对应修改位置与项目资源中已确认存在的测试/检查证据；记录命令、退出码和范围。
- 每个非问题与信息不足项必须保留可定位的反证或待补信息，不能只写结论。

## 完成 Gate

- 所有意见逐条有分类、证据和处置。
- 确认问题已修改且能定位到文件/章节/代码行。
- 受影响测试和长期文档已同步或明确不适用依据。
- 仍需业务选择的内容没有被 Agent 擅自拍板。

## 失败处理

- 评审原文或被评材料缺失：列出缺失项，停止判定对应意见。
- 代码与文档证据冲突：保留冲突两侧与影响，不能机械选一边。
- 修改引出新范围：回到用户确认，不把评审处理扩成未授权开发。
