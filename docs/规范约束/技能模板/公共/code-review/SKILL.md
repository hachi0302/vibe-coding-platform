# 审查代码

## 执行流程（Reviewer）

1. 读取需求/详设与项目规则，确认改动应解决什么问题。
2. 查看完整 diff 和调用链，不只看被点名文件。
3. 优先检查正确性：状态、金额/精度、权限、并发、幂等、异常、资源释放和兼容性。
4. 检查是否复用已有组件与公共能力，是否出现重复实现或跨层职责。
5. 检查测试能否证明行为：正常、边界、异常、原 Bug case 与关键回归。
6. 检查文档同步、敏感信息和无关改动。

## 输出契约

先列问题，按严重度排序；每项包含文件位置、触发条件、影响、证据和建议。没有问题时明确说明仍存在的测试盲区或未验证事实。

| 级别 | 判定 |
|---|---|
| 阻断 | 会造成错误结果、安全/数据风险或不可接受回归 |
| 重要 | 在明确场景下出现缺陷或维护风险 |
| 建议 | 不影响正确性但可提升清晰度 |

不要用风格偏好冒充缺陷，不要求与本任务无关的重构。

## 证据要求

每个 finding 必须包含：文件与位置、触发输入/状态、错误行为、为什么现有测试未覆盖、最小修复方向。不能指出可触发场景的问题不列为阻断项。

## 完成 Gate

- 已读需求/详设而不是只看 diff。
- 已沿入口和共享调用方检查影响面。
- 已核对项目规则、错误/权限/数据/状态约束和测试证据。
- Findings 按严重度排序且没有把推测写成事实。
- 无 finding 时仍列出未执行测试或不可访问环境造成的盲区。

## 失败处理

- 缺需求/详设：先基于明确契约审查正确性，并把无法判断的业务项列为盲区。
- diff 不完整：不对未展示代码作全量通过结论。
- 测试无法运行：保留静态 finding，并明确动态行为尚未验证。
