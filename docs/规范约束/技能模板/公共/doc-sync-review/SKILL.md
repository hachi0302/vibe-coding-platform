# 提交前文档一致性审核

## 触发条件

- 用户已明确准备提交或推送，且存在暂存区改动。
- 项目存在文档同步规则、长期文档索引或初始化生成的项目文档；没有证据时只记录缺口，不假设文档位置。

## 项目资源

| 资源 | 当前项目位置/状态 |
|---|---|
| 文档同步规则 | `{{文档同步规则路径或未配置}}` |
| 长期文档索引 | `{{文档索引路径或未配置}}` |
| 暂存区 diff 命令 | `git diff --cached --name-status`、`git diff --cached --no-ext-diff` |
| 审核记录脚本 | `{{审核记录脚本或未配置}}` |

## 执行步骤

1. 读取文档同步规则和长期文档索引；确认项目实际使用的接口、枚举、物理模型、架构、前端或集成文档。
2. 执行 `git diff --cached --name-status` 与 `git diff --cached --no-ext-diff`，以暂存区为唯一审核范围。
3. 为每个受影响领域建立“改动 → 真实长期文档 → 同步结论 → 证据”映射。
4. 仅更新真实受影响的文档并加入暂存区；无文档变化时记录项目化依据。
5. 项目存在审核记录脚本时，按该项目实际命令执行并记录结果；暂存区后续变化必须重新审核。

## 禁止事项

- 禁止因为准备提交而创建空文档、复制模板事实或编造接口/表/枚举/架构结论。
- 禁止将工作区未暂存改动纳入本次审核；禁止在用户未授权时提交、推送或修改 Git 配置。
- 禁止把“未检查”写成“不需要同步”。

## 可执行验证

- 必须记录暂存区 diff 命令、退出码、审核范围和每项文档同步结论。
- 仅运行项目资源中确认存在的审核脚本；没有脚本时输出审核表和未自动验证的原因。

## 待补信息

| 缺失信息 | 已确认事实 | 影响 | 后续证据 |
|---|---|---|---|
| {{文档同步规则/索引/审核脚本}} | {{暂存区文件列表}} | {{无法自动确认的同步项}} | {{应读取的规则、文档或用户确认}} |

## 完成 Gate

- 每个暂存区改动都已有“同步/不适用/待补信息”结论及事实依据。
- 已更新的文档与代码处于同一暂存区；未生成空壳或项目外通用文档。
- 审核记录仍对应当前暂存区指纹；若 diff 变化，必须重新执行。

## 失败处理

- 暂存区为空：报告当前没有可审核的提交范围，不擅自 `git add`。
- 文档规则或索引缺失：列入待补信息，完成代码可证明的映射，不伪造结论。
- 审核脚本失败：记录真实错误与已完成的人工审核范围，不绕过或修改脚本求通过。

## 项目专用脚本（仅在项目已安装时）
