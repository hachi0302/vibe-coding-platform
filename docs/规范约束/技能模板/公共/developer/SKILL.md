## 核心原则

以项目事实、详设和测试为约束完成最小改动。不得绕开已有框架、增加未经需求支持的兜底，或改坏无关历史逻辑。

## 执行流程（Pipeline，不可跳步）

1. 读取 `{{Agent入口}}`、详设、进度文档及本任务触发的 rules/skills。
2. 检查工作区、当前分支和用户未提交改动；需要隔离时遵循项目 Git 协作规则，并由用户选择是否使用平台提供的 Worktree 能力。
3. 沿完整链路核实现状与复用点，确认详设仍与代码一致；不一致先更新设计依据。
4. 先补失败测试：正常、边界、异常、原 Bug case；确认失败原因正是待实现行为。
5. 用项目既有分层、组件、工具类、错误体系和命名完成最小实现。
6. 跑定向测试，再跑项目约定的编译、静态检查和受影响回归测试。
7. 更新进度文档、业务总览及真实受影响的接口/模型/组件文档。
8. 审查 diff：删除临时代码，确认无无关格式化、无占位符、无敏感信息。
9. 汇报文件、测试命令与结果、风险和未完成项；是否 commit/push 由用户选择。

### Gate 1 — 开发输入完整

详设未确认、进度未建立、工作区边界不清、关键业务语义未确认时不得写业务代码。Bug 修复还必须有原 Bug 可失败用例或可复现证据。

### Gate 2 — 实现符合项目框架

每个新增文件都能说明为什么不能复用；每个修改点都能关联详设与测试。出现项目现有分层/工具/契约与详设冲突时，先修订设计依据，不私建平行框架。

### Gate 3 — 测试证明行为

定向测试、相关回归和项目构建必须给出真实命令与结果。编译通过不等于业务正确，mock 调用次数不等于真实状态/数据结果正确。

### Gate 4 — 交付由用户决策

先展示 diff 摘要、自测证据、文档同步与剩余风险，再让用户选择保留改动、提交或提交并推送。未选择时不得自动执行 Git 写操作。

## 强制边界

- 只读到代码不能证明运行时事实；日志、数据库或外部系统不可访问时如实说明。
- “加默认值继续”“吞异常”“模拟成功”“自动降级”不是修复，除非需求和历史契约明确要求。
- 不覆盖用户改动，不执行破坏性 Git 操作，不自动提交或推送。

## 完成 Gate

实现、测试、文档和交付证据全部齐全才可报告完成；编译通过不能替代行为自测。

## 失败处理

- 测试失败：定位并修复真实原因，不删除测试或放宽断言求绿。
- 构建受环境阻断：记录命令、错误和已验证范围，不声称完成。
- 发现详设遗漏：先回写详设与进度，再继续实现。
- 用户已有改动冲突：停止覆盖，说明重叠文件与可选处理方式。
