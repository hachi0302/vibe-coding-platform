### 2.4 开发后端到端真实自测（**强约束，开发完成前必填，对应轨道 B**）

> 本节对应详设模板 §9.0 **轨道 B** —— **开发完成后真实走代码自测**，测试类命名固定为 **`{需求主类}E2ETest`**（每个需求**只 1 个**，长期保留）。
>
> 历史踩坑：曾出现测试类名叫"集成测试"但实际用 fake `ProceedingJoinPoint` + 反射注入 mock Reporter 自造调用链，**根本没走真实 Spring AOP 代理 + 真实业务方法**——这种"糊弄式"自测必须杜绝。本节是「开发完成」汇报前的硬性闸门，跑不通 / 没证据 = 不允许宣告完成。

**强制要求**：开发完成后必须有 **1 个且只有 1 个** `{需求主类}E2ETest`（main + `AnnotationConfigApplicationContext` + `@EnableAspectJAutoProxy` 或 `@SpringBootTest` 等价方式），且**同时满足下面 5 条「真实」**：

1. **真实启动 Spring 上下文** — 不是 `new XxxClass()` / 反射 / 手工 `setField` 注入 fake bean
2. **真实 AOP 代理** — `@WatchHubReport` / `@Idempotent` / `@PartnerTenantScope` / `@Transactional` 等注解必须经过 Spring 真实代理生效；禁止 fake `ProceedingJoinPoint` / `Proxy.newProxyInstance` 自造调用链
3. **真实业务方法** — `DataBaoClient` / `ZthySmsClient` / `xxxService` 等真实方法被调用，不是 mock 替身
4. **真实第三方** — dev 环境真实机构 SDK / HTTP 调用（易宝沙箱 / 微信沙箱 / 真实短信通道），禁止 mock 机构响应
5. **真实落地副作用** — 上报真到 watchhub 控制台 / 数据真落 dev 库 / 真发回调 / 真生成订单

**禁止的"糊弄套路"（命中任一 = 自测无效）**：

- ❌ `Proxy.newProxyInstance(...)` 自造 `ProceedingJoinPoint` 当真实 AOP 代理
- ❌ `setField(aspect, "reporter", mockReporter)` / `ReflectionTestUtils.setField` 反射注入 mock 替换真实 Bean
- ❌ `RuntimeException → catch 后构造 success=false` 自己断言数据正确性，绕开端到端链路
- ❌ 直接 `reporter.report(...)` / `service.method(...)` 调用绕开 Aspect / Controller 入口
- ❌ 按横向维度拆多个 E2ETest（如 `*ReporterE2ETest` / `*AspectE2ETest` 分立），必须**合并到唯一的 `{需求主类}E2ETest` 内部**用方法粒度区分

**轨道 B 测试类清单（开发完成前必填，每个需求只 1 行）**：

| # | 测试类（仓库真实路径，**必须是 `{需求主类}E2ETest`**）| 真实启 Spring | 真实代理生效 | 真实业务方法 | 真实第三方 | 真实落地证据（订单号/事件 SEQ/截图描述）| 状态 |
|---|---|---|---|---|---|---|---|
| 1 | `src/test/java/.../{需求主类}E2ETest.java` | ⬜ | ⬜ | ⬜ | ⬜ | | ⏳ |

> 「开发完成」汇报前必须给出：① 轨道 B `{需求主类}E2ETest` **路径** + ② 真实跑通的**控制台 / DB / 平台侧证据**（订单号 / 事件 SEQ / 真实截图描述 / 真实落表 SQL 查询结果）。缺任一 = 不允许宣告完成。
