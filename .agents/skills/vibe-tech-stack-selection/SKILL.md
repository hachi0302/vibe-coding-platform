# Vibe Tech Stack Selection

用于 `vibe-coding-platform` 的项目工厂。目标不是推荐流行名词，而是给出能上线、能维护、可生成且可复核的技术决策。

## 决策优先级

不得颠倒以下顺序：

1. 项目阶段和业务场景：原型、正式业务、长期演进；用户、核心流程、数据一致性、性能、合规与终端形态。
2. 已有项目和团队基础设施：现有代码、数据库、云服务、运维平台、CI/CD、监控、团队技能与发布模式。
3. 用户明确偏好：只要不违背前两项，优先采用。
4. 本机已安装工具和版本：用于降低启动成本与检查兼容性，不能反过来决定架构。
5. 无证据时的保守默认：保持最少组件，并把前提写为假设。

禁止把 Java、MySQL、PostgreSQL、MyBatis-Plus、Redis、Kafka、Nacos、Docker 或任意版本当成无条件默认答案。

## 必读参考

1. 架构形态：`../software-architect/references/architectural-styles.md`。
2. 运行时：`../software-architect/references/backend-runtimes.md` 与 `references/runtime-and-frameworks.md`。
3. 数据与中间件：`../software-architect/references/data-layer.md` 与 `references/data-and-infrastructure.md`。
4. 上线基线：`references/production-baseline.md`；需要准备发布时再读取 `../production-ready/SKILL.md`，只引用规则，不执行其脚本或安装工具。
5. 可生成能力：`references/selection-matrix.md`。无法生成的方案不得伪装为可创建项目。
6. 回归样例：`references/evaluation-cases.md`。修改选型规则、提示词或 JSON 契约后必须覆盖其中至少一个不同运行时场景。

## Pipeline

### 1. 归一化需求与上下文

从一句话、产品文档或已有项目提取：项目阶段、目标用户、终端、核心业务动作、数据性质、已有服务、团队偏好、部署条件与非功能要求。

信息不足时由智能体按当前项目生成 0 到 3 个澄清问题，极端复杂场景最多 10 个；题目可以单选或多选。已从需求确定的事实不得重复询问；不影响首期选型的缺失项必须写为假设。每个问题必须给出贴合场景的推荐答案，用户未修改推荐答案时不得再次调用智能体。

### 2. 先决定结构和运行时

- 默认模块化单体或前后端分离，不预设微服务、MQ、配置中心。
- 先按工作负载与团队能力在 Java、TypeScript/Node、Python、Go、Rust、.NET 等候选中比较；语言不是模板限制。
- 运行时选择需要给出成立条件、取舍与替代方案。现有团队或项目使用的兼容技术，在满足需求时优先于理论最优。

### 3. 分项决策数据和基础设施

逐项输出以下决策；每项只能是 `adopt`、`defer` 或 `not-needed`，并有业务理由：

- 关系型/文档/键值/对象/搜索/时序数据存储。
- ORM 或数据访问方式。
- 缓存与分布式锁。
- 消息和异步任务。
- 服务发现与配置管理。
- API 协议、身份认证、文件存储。
- 可观测性、数据库迁移、备份与发布基线。

`adopt` 必须说明选择、使用场景、依赖环境和取舍；`defer` 必须说明触发条件；`not-needed` 必须说明当前为什么不引入。不能只列产品名。

### 4. 处理版本和本机环境

- 检查目标开发工具与其真实版本；只要满足所选框架兼容范围，优先复用。
- 未安装或不兼容时，才给出该项目文档维护的推荐版本、最低兼容范围和系统对应安装方式。
- 数据库、缓存、消息、配置中心通常是 Docker、测试环境、云服务或团队平台提供，不能混入“创建项目必须安装”的本机工具。
- 不能凭记忆声称版本兼容；需要时以本机检测、生成器锁定文件或官方文档为准。

### 5. 生成前 Gate

创建前必须确认：

- 推荐理由来自项目特征或明确假设，不是固定文案。
- 每个基础设施都有采用/延后/不需要的结论和理由。
- 选择的运行时、数据库驱动、ORM 与可生成模板一致；未实现的组合只允许生成实施蓝图，不能宣称可运行。
- 项目会生成 `docs/architecture/技术选型.md`，记录假设、最终方案、外部服务、版本来源、取舍与演进触发点。
- 有数据持久化时，数据库迁移、健康检查、配置密钥、日志、备份责任和测试策略均有对应基线。

## 页面和交付规则

- 推荐页按“应用运行时”“数据与存储”“异步与集成”“配置与运行”“上线基线”展示；每项显示结论和理由。
- 不展示内部评分；可以展示假设、取舍、暂不引入的原因与未来触发条件。
- 本机环境页只显示真正需要的开发工具；外部服务显示提供方式和连接待办。
- 生成的每个独立项目各自保留技术选型文档、启动说明、AI 规则和自测入口。
