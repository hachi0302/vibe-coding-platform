# 数据与基础设施决策

## 存储

| 能力 | 采用信号 | 典型选择 | 当前不采用的信号 |
| --- | --- | --- | --- |
| 事务关系数据 | 订单、账户、权限、关联查询、一致性约束 | 复用已有 MySQL/PostgreSQL；无既有约束时比较查询与扩展能力 | 纯静态站或只调用第三方 API |
| PostgreSQL 特性 | JSONB、地理、全文、向量、复杂分析，或已有 PostgreSQL 平台 | PostgreSQL | 现有 MySQL 足够且迁移没有业务收益 |
| 文档数据 | 数据天然聚合、结构频繁变化、少跨实体事务 | MongoDB | 需要大量关联、强约束或跨实体事务 |
| 对象存储 | 用户文件、图片、音视频、导入导出文件 | S3 兼容服务、云对象存储、MinIO | 仅少量静态资源且随应用发布 |
| 搜索 | 复杂全文检索、聚合、模糊匹配成为核心体验 | OpenSearch/Elasticsearch/Typesense | 数据库检索已满足规模和体验 |
| 时序数据 | 指标、设备遥测、高频时序查询 | Prometheus、TimescaleDB、InfluxDB | 普通业务审计日志 |

## 缓存与异步

- Redis：仅在热点读、会话、限流、分布式锁、排行榜、短时状态或防止缓存击穿有明确需求时采用；必须说明失效、降级和数据一致性策略。
- 不需要缓存：读取规模低、数据变化频繁、缓存会扩大一致性风险时明确写出。
- RabbitMQ：可靠异步任务、命令队列、复杂路由、延迟/重试/死信处理是核心时采用。
- Kafka：高吞吐事件流、多消费者回放、日志/埋点/流处理或跨服务事件传播明确时采用。
- RocketMQ/Pulsar：团队已有平台或其事务消息、多租户、跨地域能力是硬约束时再比较。
- 没有异步解耦、削峰、可靠事件通知或长任务时，不引入 MQ；定时任务或应用内异步即可时要写明边界。

采用任何 MQ 都必须同时写入幂等键、重试、死信、监控和消息契约版本策略。

## 配置、注册与运行

- 单体或少量服务：应用配置、环境变量、密钥管理与部署平台配置优先；不要为“标准化”引入 Nacos/Apollo。
- Nacos：服务发现、动态配置、多个 Java 服务、团队已有 Nacos 平台或 Spring Cloud 集成需求明确时采用。
- Apollo：配置审计、灰度发布和组织已使用 Apollo 的治理需求明确时采用。
- Kubernetes/云平台已经提供服务发现和配置治理时，优先复用平台，不重复建设。
- API 服务上线：必须考虑健康检查、结构化日志、关联 ID、配置与密钥隔离、数据库迁移和备份责任。
- 中大型或关键服务：补充指标、告警、追踪、容量策略、回滚与运行手册；优先融入已有观测平台。
