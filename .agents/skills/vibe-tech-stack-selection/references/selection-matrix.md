# 可生成能力与选型边界

技术分析可以比较所有常见运行时；“可创建”只表示项目工厂已经具备对应的可运行骨架、依赖配置与构建验证，不能混为一谈。

| 候选运行时 | 适配信号 | 首选框架 | 当前生成要求 |
| --- | --- | --- | --- |
| Java | 复杂事务、流程、企业集成、长期维护、团队已有 JVM | Spring Boot | 生成独立服务、健康检查、迁移与配置基线 |
| TypeScript | 前后端同语言、I/O API、快速业务迭代 | NestJS | 生成独立服务、健康检查、配置基线 |
| Python | 模型推理、数据处理、科学计算、自动化工作流 | FastAPI / Django | 生成虚拟环境、依赖锁定、健康检查 |
| Go | 网关、高并发 I/O、云原生工具、低资源服务 | chi / Gin | 生成模块、测试、健康检查、容器基线 |
| Rust | 性能/内存安全敏感服务、本机核心能力 | Axum / Tauri | 生成 Cargo 工程、测试、健康检查 |
| .NET | 团队已有 C#、Microsoft/Azure 集成 | ASP.NET Core | 生成 .NET 工程、健康检查、配置基线 |

## 结构选择

- 原型、小团队、领域未稳定：单体或模块化单体优先。
- 前端与后端独立发布、权限边界或团队协作明确：前后端分离。
- 微服务：只有独立扩缩容、稳定领域边界、服务所有权、CI/CD、监控与故障处置能力同时具备时才采用。
- monorepo 是代码组织方式，不等同于微服务；仅在共享类型、组件或多应用协同收益明显时采用。

## 版本规则

- 不在选型矩阵中固定框架、JDK、Node、数据库或中间件版本。
- 创建前先检测本机兼容版本；缺失时由生成器维护的项目文档提供推荐版本与安装方式。
- 已有项目优先读取其锁定文件、构建文件、容器和部署清单，再判断能否升级。
