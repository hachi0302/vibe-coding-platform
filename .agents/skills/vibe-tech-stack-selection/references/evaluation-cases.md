# 选型回归场景

用于检查智能体是否真正按项目决策，而非按关键词或固定模板输出。每个场景都要求给出采用、延后或不采用的理由。

## 1. 内部订单后台，Java 团队，已有 MySQL 和 Nacos

- 预期：可以选择 Java/Spring Boot 与已有 MySQL；是否使用 Nacos 取决于服务数量和现有平台，不因为存在就强制新增。
- 不应：默认 Kafka、Redis、微服务；不能把 PostgreSQL 当成必须替换项。

## 2. AI 文档分析产品，Python 团队，调用模型并处理异步文件

- 预期：Python/FastAPI 可优先；对象存储在文件持久化存在时采用；异步任务按耗时和可靠性决定队列。
- 不应：因为“正式项目”就改成 Java；没有吞吐或回放需求时不默认 Kafka。

## 3. 需要 SEO 的产品官网，只有表单和内容管理

- 预期：服务端渲染框架优先；数据库只在内容/表单持久化需要时采用。
- 不应：强行生成独立 Java 后端、缓存、MQ 或配置中心。

## 4. 高并发协议网关，团队有 Go 运维经验

- 预期：Go 可以优先；关系库、缓存、MQ 仅按真实连接状态、路由、审计和异步需求决定。
- 不应：因公司其他项目使用 Java 就忽略工作负载和团队能力。

## 5. 已有 Python 服务改造，Docker/Kubernetes 提供配置和服务发现

- 预期：优先复用 Python 与平台配置；不重复引入 Nacos/Apollo。
- 不应：仅因本机装有 JDK 或 Maven 就迁移到 Java。
